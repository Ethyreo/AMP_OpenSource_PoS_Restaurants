# App Plan and Requirements

**Captured:** 2026-03-23  
**Source:** User-supplied product direction before implementation

## Product Direction

Build a restaurant PoS experience with a Cred-style aesthetic: calm screens, strong whitespace, premium typography, muted pastel accents, subtle depth, and minimal visual noise during busy shifts.

## Visual Language

### Color Palette

| Element | Tone | Hex |
|---------|------|-----|
| Primary Background | Off-White / Soft Bone | `#F7F8FA` |
| Card Background | White | `#FFFFFF` |
| Accent (Veg) | Muted Sage Green | `#A3B18A` |
| Accent (Non-Veg) | Dusty Rose / Terra Cotta | `#E07A5F` |
| Action Buttons | Deep Slate | `#2D3142` |
| Secondary Text | Cool Grey | `#94A3B8` |

### UI Rules

- Use modern sans-serif typography with clean geometry.
- Keep headings bold but compact.
- Use large card radii in the 16px to 24px range.
- Use soft shadows and subtle depth instead of hard borders.
- Use glassmorphism selectively for bottom navigation or billing summaries.

## Local Architecture Direction

- Use Antigravity IDE with a local web-style app architecture.
- Keep logic strictly local on the device.
- Use IndexedDB instead of LocalStorage for structured persistence.
- Keep the floor state in local JavaScript objects and repositories.
- Use smooth drawers and transitions instead of popup-heavy flows.

## Core Workflow

1. Floor view shows a minimalist grid of tables.
2. Occupied tables use a subtle pulse state.
3. Selecting a table opens a slide-up order sheet.
4. Billing recalculates instantly as items change.
5. Every order mutation auto-saves to IndexedDB.
6. The app can generate bills locally and send clean output to print.

## Billing Requirements

The total payable is calculated as:

`P = round((S - D) + SC + GST)`

Where:

- `S` = subtotal
- `D` = discount
- `SC = (S - D) x 0.05` or configured rate
- `GST = (S - D + SC) x 0.05` or configured rate

### Billing Rules

- Tax logic is hardcoded initially but editable through Settings.
- Service charge percentage must be configurable.
- GST must be configurable.
- Billing output must stay consistent across screen, PDF, and print.

## Offline Persistence

- Auto-save every item addition or change into IndexedDB.
- Recover table state after app crash or device power loss.
- Keep final billing history locally.

## Feature Modules

- Quick Search by code or name
- Floor View
- Order Sheet drawer
- Billing Engine
- Daily Summary
- Table Switching via long press
- Local Printing and PDF generation

## Reporting

The End of Day screen should show:

- total revenue
- most sold items
- payment mode split such as UPI and cash

## Implementation Notes To Preserve

- Define CSS variables globally in `:root`.
- Use print-specific CSS to hide navigation and controls from receipts.
- Keep logic local-first and avoid server dependence for core flows.

## Critical Clarification

This plan describes a local web app architecture running on Android devices. If the intention is a native Android app, the architecture should be revisited before implementation.
